# Code comments

## Language

Comments: **English**, always. (Code — variables, functions, classes, constants — is English too; that lives in `## Language` of the development fragment.)

## Base rule — default: NO comment

A comment is valid **only if it explains the WHY** (intent, constraint, tradeoff). **Never** the WHAT or HOW.

**10-second test:** if a competent reader gets the same from the code in under 10s, the comment is redundant → **delete it**.

## Conciseness

If a comment is valid, keep it **one line**. A comment that needs a paragraph is a signal the WHY belongs somewhere more durable — an EDR, the commit body, or a test name — not inline. Length is not thoroughness; a verbose comment rots faster and is read less.

## Before commenting, check the WHY doesn't belong elsewhere

A comment is the last resort, not the first. Before writing one, ask if the WHY is better expressed as:

- A **better name** (`retryAfter429` beats `// wait before retrying`).
- A **test** whose name states the expected behavior.
- The **commit body** (see the `git` skill) — motivation, evaluated alternatives.
- An **EDR** — an architectural decision with reach beyond this line.

Only when none of those fits does the comment earn its place.

## Allowed exceptions

- Rationale for a non-trivial algorithm.
- A design decision not deducible from the code.
- A workaround (with a link to the issue/cause).
- A warning about a non-obvious consequence.

## Good examples (valid WHY)

```
// O(n) on purpose: n < 16 here, a hash map is slower for this size.
total = items.reduce(sum)

// Stripe rounds half-up; we must match or reconciliation breaks. See FIN-204.
amount = roundHalfUp(raw)

// WARNING: must run before auth middleware — it seeds the request context.
app.use(contextSeed)
```

Each says something the code cannot. None paraphrases it.

## Public-API doc-comments vs noise

Documenting the **contract** of an exported symbol is NOT the same as a redundant docstring.

- **Valid** — a doc-comment on a public/exported symbol (godoc, rustdoc `///`, public JSDoc) that states contract a caller needs: invariants, units, error/nil behavior, ownership.
- **Forbidden** — an autogenerated or restating docstring (`// GetUser returns the user` over `getUser()`). It adds nothing the signature doesn't.

Rule of thumb: document the **contract**, never the **implementation**. Internal/private symbols rarely need a doc-comment at all.

## Forbidden

- Paraphrasing the code.
- Section-title comments.
- Autogenerated / restating docstrings.
- Commented-out code.
- `TODO`/`FIXME` without actionable context.

## Anti-patterns

| ❌ Bad | ✅ Good |
|--------|--------|
| `// increment counter by one` over `counter++` | (no comment) |
| `// --- helpers ---` | (no comment; let structure speak) |
| `// loop over users` over an obvious `for` | (no comment) |
| `// TODO: fix this` | `// TODO(#412): retry on 429 once rate-limit headers land` |
| `// returns the user` over `getUser()` | (no comment) |
| old code left between comments | delete it (git remembers) |

## Project convention precedence

If the repo enforces its own convention via an EDR or tooling — license headers, mandatory public JSDoc, doc-comment linting (godoc, rustdoc) — **that convention wins** over this skill for the cases it covers. This skill is the default, not an override of an explicit project decision.

## Self-check (before writing any comment)

1. Does it explain **WHY**, not WHAT/HOW? If not → don't write it.
2. Would a competent reader get it from the code in <10s? If yes → don't write it.
3. Does the WHY belong in a name / test / commit / EDR instead? If yes → put it there.
4. Can it be **one line**? If not → reconsider where it belongs.
