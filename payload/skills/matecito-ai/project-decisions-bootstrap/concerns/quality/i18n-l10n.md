# Fase: Internacionalización y localización

## Qué decide

Si la aplicación soporta múltiples idiomas o configuraciones regionales, qué librería gestiona las traducciones, y dónde viven los strings.

## Preguntas

### 1. Alcance de idiomas y formatos regionales

> Agregar i18n después de que la app está construida es costoso — los strings están dispersos en el código. Definirlo ahora, aunque sea "solo un idioma", establece la estructura que permite crecer.

- **Un solo idioma, sin i18n formal** — *default para proyectos internos o MVPs sin requerimiento multiidioma; igual usar una constante de idioma para no hardcodear strings en lógica.*
- Múltiples idiomas — indicar cuáles (ej: `es`, `en`, `pt`); fallback si un string no existe en el idioma pedido.
- Un idioma ahora, preparar estructura para agregar más después.
- No sé, recomendame.

### 2. Dónde viven las traducciones y qué librería

> **Solo si eligió múltiples idiomas o preparar estructura.** El formato y la librería determinan cómo se integra i18n con el framework.

- **Archivos JSON / YAML por idioma** (`locales/en.json`, `locales/es.json`) con librería del ecosistema — *default: `i18next` para JS/TS, `django.utils.translation` para Django, `go-i18n` para Go, `java.util.ResourceBundle` para Java.*
- Archivos `.po` / `.mo` (gettext) — estándar en ecosistemas Python/PHP; herramientas de traducción ampliamente soportadas.
- Plataforma externa de gestión de traducciones (Crowdin, Lokalise, Phrase) — para equipos con traductores no técnicos.
- No sé, recomendame.

## Notas de lógica (para el motor)

- Si elige "Un solo idioma, sin i18n formal", no hacer la pregunta 2. El ADR igual documenta la convención mínima: los strings de UI van en constantes, no hardcodeados en lógica.

## Tech a registrar

Si se elige una librería de i18n (ej: `i18next.md`, `go-i18n.md`), registrarla en `tech/`.

## Qué materializar

ADR `i18n-l10n` con: idiomas soportados (o decisión explícita de no soportar i18n), librería y formato de archivos de traducción, estructura de carpetas de locales, convención de claves (ej: `namespace.componente.string`), formato de fechas y números por locale, y regla de fallback si un string no existe en el idioma pedido.
